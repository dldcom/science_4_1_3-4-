# 마그마·현무암·화강암 학습 웹앱 개발 기록

작성일: 2026-06-14

## 1. 프로젝트 목표

초등학교 4학년 학생이 마그마가 식는 장소와 속도에 따라 현무암과
화강암이 만들어지는 과정을 관찰하는 웹앱을 제작한다.

현재 작업한 범위는 전체 웹앱 중 첫 장면인 **화산과 용암 장면
프로토타입**이다.

프로토타입의 목적은 다음 두 가지를 미리 확인하는 것이다.

1. 브라우저에서 현실감 있는 화산과 용암을 표현할 수 있는가?
2. 학교 태블릿에서도 실행 가능한 수준으로 만들 수 있는가?

현재 파일은 최종 수업용 앱이 아니라, 시각 품질과 성능을 검증하기 위한
중간 결과물이다.

---

## 2. 교과서 내용 확인

제공된 과학 교과서 PDF에서 화강암과 현무암 차시를 확인했다.

- 관련 차시: 교과서 76쪽~77쪽
- 선수 학습: 화산과 화산 분출물 72쪽~73쪽

교과서 표현을 기준으로 정리한 핵심 개념은 다음과 같다.

| 구분 | 화강암 | 현무암 |
| --- | --- | --- |
| 만들어지는 장소 | 땅속 깊은 곳 | 지표 가까이 |
| 식는 속도 | 천천히 | 빠르게 |
| 색깔 | 밝음 | 어두움 |
| 알갱이 크기 | 큼 | 작음 |
| 구멍 | 주요 분류 특징이 아님 | 가스가 빠져나가 생기기도 함 |

웹앱에서는 현무암의 구멍을 항상 존재하는 특징처럼 표현하지 않고,
`생기기도 한다`고 설명해야 한다.

## 3. 전체 웹앱 구상

현재까지 합의한 전체 흐름은 다음과 같다.

```text
도입
→ 화산과 용암 관찰
→ 지표 가까이에서 빠르게 냉각
→ 현무암 생성 관찰
→ 땅속 깊은 곳에서 천천히 냉각
→ 화강암 생성 관찰
→ 실제 암석 3D 모델 관찰
→ 현무암과 화강암 비교 및 분류
→ 확인 퀴즈
```

화강암:

관찰용으로 사용하고, 암석 생성 과정은 Three.js로 직접 구현하는 방향이다.

## 4. 화산 제작 방식 결정

처음에는 외부에서 완성된 화산 3D 모델을 구해 사용하는 방식을 조사했다.

### 조사한 Sketchfab 후보

| 후보 | 면 개수 | 판단 |
| --- | ---: | --- |
| volcano by gelmi.com.br | 157,203 | 시각 품질은 좋지만 태블릿 예산보다 무거움 |
| Volcano V1 by JayT3D | 80,000 | 품질과 성능 균형이 좋지만 용암이 이미 모델에 고정됨 |
| volcano with magma | 24,900 | 가볍지만 작은 전시 모형처럼 보임 |
| Erupting volcano | 22,402 | 가볍지만 표현이 다소 단순함 |

Sketchfab 모델은 `CC Attribution` 라이선스였으며 다운로드 가능 상태였다.
그러나 다음 문제가 있었다.

- Sketchfab 다운로드 API 사용에는 별도 Sketchfab 인증이 필요함
- 이미 만들어진 용암 위치와 형태를 바꾸기 어려움
- 지하 단면과 냉각 과정을 표현하기 어려움
- 원본 모델을 다시 최적화해야 함
- 저작자 표시와 라이선스 관리가 필요함

따라서 첫 프로토타입은 다음 조합으로 결정했다.

```text
화산의 큰 모양
→ Three.js 코드로 직접 생성

화산의 현실적인 표면
→ Poly Haven CC0 PBR 텍스처 사용

용암 흐름과 냉각
→ Three.js 셰이더와 ambientCG PBR 텍스처 결합
```

## 5. 웹에서 텍스처를 찾은 과정

### 5.1 조사 원칙

텍스처를 찾을 때 다음 조건을 우선했다.

1. 라이선스가 명확할 것
2. 교육용 웹사이트에 재배포할 수 있을 것
3. PBR 텍스처 세트가 제공될 것
4. 1K 해상도를 제공할 것
5. 타일처럼 반복 가능한 재질일 것
6. 웹과 태블릿에 사용하기 적절한 용량일 것

우선 조사한 사이트는 다음과 같다.

- Poly Haven: 모든 에셋이 CC0
- ambientCG: 모든 에셋이 CC0
- Sketchfab: 모델마다 라이선스가 다르므로 개별 확인 필요

CC0는 저작권자가 가능한 권리를 포기한 라이선스로, 수정과 재배포가
자유롭다. 출처 표시는 필수는 아니지만 이 프로젝트에서는 기록을 남긴다.

### 5.2 API를 사용한 검색

웹페이지에서 눈으로 검색하는 것 외에도 공개 API를 사용해 에셋을 조사했다.

Poly Haven 텍스처 목록:

```text
https://api.polyhaven.com/assets?t=textures
```

Poly Haven 개별 파일 정보:

```text
https://api.polyhaven.com/files/{asset-id}
```

ambientCG 검색:

```text
https://ambientcg.com/api/v2/full_json?type=Material&q=lava
```

ambientCG 개별 에셋 정보:

```text
https://ambientcg.com/api/v2/full_json?id=Lava002&include=downloadData
```

이 API 응답에서 다음 내용을 확인했다.

- 에셋 이름과 ID
- 라이선스
- 미리보기 이미지 주소
- 제공되는 PBR 맵 종류
- 해상도별 다운로드 주소
- 압축 파일 용량

검색 결과만 보고 선택하지 않고, 후보 미리보기 이미지를 모두 내려받아
육안으로 비교했다.

## 6. 화산 표면 텍스처

### 선택한 재질

- 이름: Gray Rocks
- 출처: https://polyhaven.com/a/gray_rocks
- 라이선스: CC0
- 사용 해상도: 1K

### 사용한 파일

```text
gray_rocks_diff_1k.jpg
gray_rocks_nor_gl_1k.jpg
gray_rocks_rough_1k.jpg
```

### 각 파일의 역할

#### Diffuse / Base Color

실제 눈에 보이는 암석 색상과 무늬를 담는다.

```javascript
material.map = rockColor;
```

#### Normal Map

실제 3D 모양을 추가하지 않고, 빛이 표면에서 반사되는 방향을 바꿔
작은 돌과 홈이 있는 것처럼 보이게 한다.

```javascript
material.normalMap = rockNormal;
```

#### Roughness Map

표면 위치별로 얼마나 반짝이거나 거칠게 보일지를 결정한다.

```javascript
material.roughnessMap = rockRough;
```

### Three.js에서 텍스처를 불러온 방법

```javascript
const loader = new THREE.TextureLoader();

const [rockColor, rockNormal, rockRough] = await Promise.all([
  loader.loadAsync("./assets/gray_rocks_diff_1k.jpg"),
  loader.loadAsync("./assets/gray_rocks_nor_gl_1k.jpg"),
  loader.loadAsync("./assets/gray_rocks_rough_1k.jpg"),
]);
```

텍스처가 지형 전체에 한 번 크게 늘어나지 않도록 반복 설정을 적용했다.

```javascript
for (const texture of [rockColor, rockNormal, rockRough]) {
  texture.wrapS = texture.wrapT = THREE.RepeatWrapping;
  texture.repeat.set(3.2, 3.2);
}
```

모든 반복 타일은 실제로 동일하다. 그러나 반복이 잘 보이지 않는 이유는
다음과 같다.

- 텍스처가 가장자리가 연결되는 seamless 방식으로 제작됨
- 아래에 있는 실제 3D 지형 모양이 위치마다 다름
- 표면 방향에 따라 조명이 다르게 적용됨
- Normal Map이 미세한 빛 반사를 바꿈
- 카메라가 텍스처 반복을 발견하기 어려운 거리에서 관찰함

### 화산 재질 생성

```javascript
const terrainMaterial = new THREE.MeshStandardMaterial({
  map: rockColor,
  normalMap: rockNormal,
  roughnessMap: rockRough,
  roughness: 0.92,
  color: 0x4c4541,
});
```

## 7. 화산 지형 생성 과정

화산은 완성된 3D 모델을 사용하지 않고 평평한 격자를 변형하여 만들었다.

### 7.1 평평한 지형 격자 생성

```javascript
const terrainGeo = new THREE.PlaneGeometry(78, 78, 112, 112);
terrainGeo.rotateX(-Math.PI / 2);
```

112개의 가로·세로 구간을 사용한다. PC에서도 태블릿과 같은 설정을
사용한다.

### 7.2 각 꼭짓점 높이 계산

```javascript
for (let i = 0; i < positions.count; i++) {
  const x = positions.getX(i);
  const z = positions.getZ(i);
  positions.setY(i, terrainHeight(x, z));
}
```

`terrainHeight(x, z)` 함수는 다음 요소를 합쳐 해당 지점의 높이를 결정한다.

```text
기본 화산 원뿔
+ 중앙의 분화구
+ 산비탈 능선
+ 부드러운 노이즈 기반 침식 표현
= 최종 높이
```

작은 돌 하나하나는 실제 3D로 만들지 않고 PBR Normal Map으로 표현한다.
큰 실루엣과 골짜기만 실제 지형 높이로 만든다.

## 8. 용암 PBR 텍스처 조사

### 조사 결과

ambientCG에서 `Lava001`부터 `Lava005`까지 다섯 개의 CC0 용암 PBR
재질을 찾았다.

| 후보 | 판단 |
| --- | --- |
| Lava001 | 밝은 틈과 흐르는 주름이 강함 |
| Lava002 | 검게 식은 껍질과 뜨거운 틈의 균형이 가장 좋음 |
| Lava003 | 뜨거운 부분이 더 많음 |
| Lava004 | 거의 완전히 녹은 주황색 표면 |
| Lava005 | 따뜻한 주름 중심 |

최종적으로 `Lava002`를 선택했다.

- 출처: https://ambientcg.com/view?id=Lava002
- 라이선스: CC0

### Lava002 1K JPG 파일 구성

| 파일 | 용량 | 사용 여부 |
| --- | ---: | --- |
| Color | 약 1.67MB | 사용 |
| Emission | 약 1.44MB | 사용 |
| NormalGL | 약 2.28MB | 사용 |
| Roughness | 약 0.31MB | 사용 |
| Displacement | 약 0.40MB | 사용하지 않음 |

Displacement는 실제 메시 표면을 변형한다. 좁은 용암 줄기에서는 효과에
비해 성능 비용이 크므로 제외했다.

### 용암에서 Emission 맵이 중요한 이유

Emission 맵은 빛나야 하는 영역을 지정한다.

```text
검은 영역
→ 식은 껍질, 빛나지 않음

밝은 영역
→ 뜨거운 틈, 스스로 빛남
```

따라서 실제 검은 용암 껍질을 유지하면서 틈 사이의 뜨거운 용암만
강조할 수 있다.

## 9. 용암 셰이더와 PBR 결합 방식

처음 용암은 단순한 주황색 리본이었다. 이후 GPU가 실시간으로 용암
무늬를 계산하는 셰이더를 만들었다.

셰이더는 다음 정보를 매 프레임 계산한다.

```text
현재 용암 표면 위치
+ 흐른 시간
+ 분화구에서 내려온 거리
+ 냉각 정도
+ Lava002 PBR 텍스처
= 현재 픽셀의 색상
```

### 분화구 근처

분화구에서 막 나온 용암은 매우 뜨거우므로 Lava002의 검은 껍질을 거의
사용하지 않는다.

```text
밝은 노란색과 붉은색
+ 움직이는 절차적 무늬
```

### 산 아래쪽

용암이 산 아래로 내려갈수록 Lava002 텍스처를 더 많이 섞는다.

```text
분화구에서 멀어진 거리 증가
→ Lava002 Color와 Normal 영향 증가
→ 검은 껍질과 주름이 나타남
```

### 표면 식히기 버튼

`표면 식히기` 버튼을 누르면 거리와 관계없이 모든 흐름과 분기 전체에
강한 냉각이 적용된다.

```text
뜨거운 용암
→ 검은 Lava002 껍질 증가
→ Emission 맵의 붉은 틈만 남음
```

## 10. 자연스러운 용암 경로 생성

### 초기 문제

처음에는 용암 경로를 사람이 좌표로 지정했다.

```text
분화구 → 중간점 → 중간점 → 산 아래
```

이 방식은 다음 문제가 있었다.

- 용암이 미끄럼틀처럼 매끄럽게 내려옴
- 지형의 골짜기를 따르지 않음
- 폭이 일정함
- 가장자리가 지나치게 매끄러움

### 현재 방식: 경사 추적

앱이 시작될 때 용암 경로를 한 번 계산한다.

```text
현재 위치 주변의 여러 방향 높이 측정
→ 가장 낮아지는 방향 선택
→ 조금 이동
→ 다시 주변 높이 측정
→ 반복
```

방향을 선택할 때 다음 요소를 함께 사용한다.

```text
아래로 내려가는 정도
+ 이전 진행 방향 유지
+ 산 바깥쪽으로 향하는 정도
+ 약간의 노이즈
```

완전히 가장 낮은 방향만 고르면 경로가 지나치게 꺾일 수 있기 때문에
이전 진행 방향과 부드럽게 혼합한다.

### 폭 계산

용암 폭은 위치마다 다르게 계산한다.

```text
경사가 급함
→ 용암 폭이 좁아짐

경사가 완만함
→ 용암 폭이 넓어짐

좌우 노이즈
→ 양쪽 가장자리가 서로 다르게 흔들림
```

현재 전체 폭은 초기 프로토타입보다 약 2.5배 확대했다.

### 지형 밀착

용암 리본 전체를 한 높이에 놓지 않고, 좌우 꼭짓점 각각에서 지형 높이를
다시 계산한다.

```javascript
leftY = terrainHeight(leftX, leftZ);
rightY = terrainHeight(rightX, rightZ);
```

따라서 용암이 지형 굴곡을 따라 기울고 휘어진다.

### 분기

주 흐름 중간에서 작은 분기 하나를 만들었다. 분기는 새로운 흐름처럼
계산하지만, 분기되기 전까지 이미 이동한 거리를 냉각 셰이더에 전달한다.

초기에는 분기 UV가 다시 0부터 시작하여 분기 흐름이 분화구 근처처럼
지나치게 뜨겁게 보이는 문제가 있었다.

이를 `uCoolingOffset`으로 수정했다.

```text
분기 자체 거리
+ 분기되기 전까지 이동한 거리
= 실제 냉각 거리
```

## 11. 태블릿 성능 전략

사용자 요청에 따라 PC에서도 태블릿이 처리 가능한 수준 이상의 그래픽을
사용하지 않는다.

현재 PC와 태블릿은 동일한 품질 설정을 사용한다.

| 항목 | 현재 설정 |
| --- | ---: |
| 화산 지형 분할 | 112 × 112 |
| 화면 삼각형 수 | 약 25,000개 |
| 드로콜 | 약 7개 |
| 연기 입자 | 90개 |
| 최대 픽셀 비율 | 1.25 |
| 용암 경로 계산 | 앱 시작 시 한 번 |
| 실시간 유체 물리 | 사용하지 않음 |

### 태블릿에도 GPU가 있는가?

태블릿에는 보통 CPU와 같은 칩에 들어 있는 통합 GPU가 있다.

- iPad: Apple A/M 시리즈 GPU
- 갤럭시 탭: Adreno GPU 등
- 저가형 태블릿: Mali 계열 GPU 등

Three.js는 브라우저 WebGL을 통해 GPU를 사용한다.

### 성능을 아끼는 핵심 원칙

자연스러운 용암 경로와 폭은 앱 시작 시 CPU가 한 번 계산한다.

```text
초기 한 번:
지형 경사 탐색 + 메시 생성

매 프레임:
용암 빛과 텍스처 흐름만 계산
```

실시간 유체 시뮬레이션은 사용하지 않는다.

## 12. 로컬 실행 방법

`index.html`을 더블클릭하면 브라우저 보안 정책 때문에 CORS 오류가 발생한다.

```text
file:///.../index.html
```

JavaScript 모듈과 텍스처 파일은 HTTP 웹서버를 통해 열어야 한다.

프로젝트 폴더의 다음 파일을 실행한다.

```text
실행하기.bat
```

브라우저 주소:

```text
http://127.0.0.1:4173
```

검은 서버 창을 닫으면 웹앱도 종료된다.

## 18. 용암 폭·화산 가스·현무암 표면 개선

### 용암 폭과 갈래

- 네 갈래에 같은 폭을 쓰지 않고 서로 다른 폭 배율을 적용했다.
- 정면의 주 용암은 가장 넓게 만들어 시각적 중심이 되도록 했다.
- 보조 용암 두 갈래와 작은 분기 흐름은 점차 좁게 설정했다.
- 기존의 지형 추적 경로와 좌우 폭 노이즈는 유지하므로, 넓어져도 직선 미끄럼틀처럼 보이지 않는다.

### 화산 가스와 기포

- 용암 경로 여러 지점에서 작은 밝은 기포가 맥동한다.
- 기포 지점에서 반투명 가스 입자가 위로 올라가며 옆으로 조금 퍼진다.
- 냉각이 진행될수록 기포와 가스가 줄어들고, 완전히 식으면 거의 사라진다.
- 분화구 연기를 90개에서 64개로 줄이고 용암 가스 입자 36개를 추가했다. 움직이는 입자 총량은 100개로 유지했다.

### 현무암 PBR 텍스처

- ambientCG의 CC0 `Rock041`을 현무암 표면 후보로 선정했다.
- 절벽 방향성이 강한 Rock031, Rock033, Rock035, Rock037보다 작은 표면 무늬가 반복 적용에 적합했다.
- 웹에서는 1K Color, NormalGL, Roughness 세 파일만 사용한다.
- 냉각 후반부터 Rock041이 용암 표면에 섞이고, 완전히 식으면 어두운 현무암 표면이 된다.
- 일부 기공은 별도 3D 구멍 대신 셰이더의 반복 가능한 해시 무늬로 표현했다.

### 태블릿 성능 원칙

- 유체 시뮬레이션은 사용하지 않는다.
- 기포와 가스는 각각 하나의 Points 묶음으로 그려 드로우콜을 억제한다.
- 최종 장면은 약 25,400 triangles, 9 draw calls 수준이다.
- PC에서도 태블릿과 같은 품질 설정을 사용한다.

## 19. 반복 가능한 단발 분출 구조

### 분출 한 번의 생명 주기

1. 처음에는 용암 없이 약한 분화구 연기만 보인다.
2. `분출 시작하기`를 누르면 분화구에서 폭발 입자가 솟아오른다.
3. 버튼을 누른 순간에만 3~5개의 무작위 시작 방향을 뽑고 지형 경사를 따라 경로를 계산한다.
4. 용암 리본 전체를 한 번에 보이지 않고, 인덱스 draw range를 늘려 위에서 아래로 흐르는 것처럼 보이게 한다.
5. 용암이 도착한 뒤 지난 시간에 따라 `Lava004 → Lava002 → Rock041 현무암` 순서로 자동 전환한다.
6. 모든 용암이 굳은 뒤에만 다음 분출 버튼을 다시 누를 수 있다.

### 기존 현무암 유지와 성능

- 두 번째 분출부터도 이전 현무암은 삭제하지 않는다.
- 완전히 굳은 분출의 여러 용암 리본은 하나의 누적 현무암 지오메트리로 병합한다.
- 지오메트리 병합은 삼각형 메모리를 크게 줄이지는 않지만, 오래된 현무암의 드로우콜을 한 번으로 유지한다.
- 새 분출이 흐르는 동안에만 새 용암 갈래별 드로우콜이 잠시 추가된다.
- 누적 현무암이 안전 삼각형 기준에 도달하면 새 분출을 막아 태블릿이 갑자기 멈추는 것을 방지한다.

### 조작 변경

- 기존 `빠르게 식히기` 버튼은 제거했다.
- `시간 빠르게 보기`는 전체 분출·흐름·냉각 시간을 3배 빠르게 진행한다.
- 분출은 계속 공급되는 유체가 아니라 버튼 한 번당 유한한 양만 생성된다.

### 용암 전면과 냉각 방향 수정

- 초기 구현은 완성된 삼각형 조각을 하나씩 표시해 용암 전면이 계단식으로 움직였다.
- 전체 용암 리본은 항상 렌더링하고, 셰이더의 `uReveal` 위치보다 앞쪽 픽셀을 숨기는 방식으로 교체했다.
- 전면 위치는 매 프레임 연속적인 실수 값으로 이동하므로 용암이 부드럽게 내려간다.
- UV 기준 `0`은 분화구, `1`은 산 아래쪽이다.
- 기존에는 분화구 쪽의 냉각 나이를 크게 계산해 위쪽부터 굳는 오류가 있었다.
- 냉각 나이에 아래쪽 거리 값을 더하도록 수정해 분화구에서 먼 부분부터 Lava002와 현무암으로 변한다.
